# Idle Mining Game — требования к проекту

## 1. Общая концепция

Игра является idle mining / exploration проектом, вдохновлённым Deep Corp и Gold & Goblins. Игрок исследует подземные уровни, добывает ресурсы, нанимает и объединяет работников, выполняет заказы, расширяет склады и постепенно спускается глубже.

Каждый уровень представляет собой бесконечную процедурную карту, генерируемую по seed-фразе чанками. Игрок начинает в пустой области 3×3 клетки, постепенно разрабатывает соседние клетки и ищет пустую область спуска 3×3, ведущую на следующий уровень.

## 2. Обязательное архитектурное правило

Проект должен быть разделён минимум на два крупных слоя:

1. **Game Engine / Core Simulation**
   Отвечает за всю игровую логику:

   - генерацию уровней и чанков;
   - состояние клеток;
   - добычу ресурсов;
   - работников;
   - склады;
   - заказы;
   - смены;
   - tick-систему;
   - сохранение/экспорт состояния;
   - валидацию команд.

2. **Game Render / UI Layer**
   Отвечает только за отображение и пользовательский ввод:

   - рендер карты;
   - панели UI;
   - визуализацию работников;
   - визуализацию ресурсов, складов, заказов;
   - вызов команд engine.

UI/render-слой не имеет права напрямую менять состояние игры.

Всё взаимодействие с engine происходит только через два метода:

```ts
engine.apply(command);
engine.read(query);
```

или их эквиваленты в Rust.

`apply` используется для изменения состояния.
`read` используется только для чтения состояния.
Любые обходные пути запрещены.

## 3. Модель времени

Игра использует дискретное время.

- 1 tick = 1 секунда.
- 1 смена = 300 ticks = 5 минут.
- Прогресс игры идёт только через явный вызов `tick`.
- Offline-прогресс для MVP не нужен.

Команды времени:

```ts
apply({ type: 'tick', ticksPassed });
apply({ type: 'fast_forward_to_shift_end' });
apply({ type: 'start_next_shift' });
```

Один вызов `tick` не должен перескакивать через несколько смен. Если до конца смены осталось меньше тиков, чем передано в `ticksPassed`, engine обрабатывает только остаток до конца смены и возвращает `remainingTicks`.

После завершения смены engine переходит в фазу планирования.

```ts
GamePhase =
  | "shift_running"
  | "shift_planning"
```

Во время `shift_planning` игровое время остановлено. Следующая смена начинается только после явной команды игрока.

## 4. Карта и генерация

Карта уровня бесконечная и лениво генерируемая по чанкам.

- Размер чанка для MVP: 32×32 клетки.
- Уровень генерируется по seed-фразе.
- Для каждого уровня используется `seedPhrase + levelDepth + chunkX + chunkY + generatorVersion`.
- При создании уровня сразу генерируется область 5×5 чанков.
- В центральном чанке находится стартовая область 3×3.
- В одном из нецентральных чанков области 5×5 находится спуск 3×3.
- Спуск не виден сразу, пока не попадает в радиус разведки.
- Спуск является пустой областью, к которой нужно пробуриться.

Генератор обязан гарантировать, что от стартовой области существует потенциальный путь к зоне спуска.

## 5. Типы клеток

```ts
CellKind =
  | "empty"
  | "deposit"
  | "obstacle"
  | "stairs_area"
```

- `empty` — пустота, проходимая клетка.
- `deposit` — клетка с породой и/или ресурсами.
- `obstacle` — неразрушаемое препятствие.
- `stairs_area` — часть пустой области спуска 3×3.

Пропастей нет.

После полной выработки всех компонентов `deposit`-клетка становится пустой и проходимой.

## 6. Видимость и доступность

Игрок видит состав клеток на расстоянии 5 клеток от каждой открытой/достижимой пустой клетки.

Нужно разделять:

```ts
CellVisibility =
  | "unknown"
  | "scouted"

CellAccessibility =
  | "unreachable"
  | "reachable"
```

- `scouted` означает, что игрок видит клетку и её состав.
- `reachable` означает, что к клетке есть путь от входа уровня.
- Пустоты становятся доступными только когда к ним появляется путь.
- Если игрок пробурился к связной пустой области, вся эта область открывается сразу.
- Если пустота или зона видимости выходит за границу чанка, engine должен догенерировать соседний чанк.
- Если после генерации нового чанка область снова уходит за границу, процесс повторяется.

Генератор должен избегать больших пещер. Пустоты допустимы, но большие связные пустые области нежелательны.

## 7. Компоненты клетки

Клетка может содержать смесь компонентов:

```ts
CellComponent {
  type: "resource" | "rock"
  resourceId?: ResourceId
  initialAmount: number
  remainingAmount: number
  ratio: number
}
```

Порода является обычным компонентом клетки, но не попадает на склад.

Пример:

```txt
Клетка:
- rock: 60%
- stone: 25%
- copper: 10%
- gold: 5%
```

Работники вырабатывают компоненты равномерно. Ресурсы идут на соответствующие склады. Порода просто исчезает.

Если склад для конкретного ресурса заполнен, этот ресурс временно не добывается и остаётся в клетке. Остальные доступные компоненты продолжают вырабатываться. Если остались только ресурсы, для которых нет места на складах, работники простаивают.

Клетка считается полностью выработанной только когда все компоненты имеют `remainingAmount = 0`.

## 8. Работники

Работники являются глобальным пулом игрока. Любого свободного работника можно отправить на любой открытый уровень.

Работник имеет уровень. Чем выше уровень, тем выше скорость разработки.

Merge:

```txt
2 работника одного уровня → 1 работник следующего уровня
```

Merge возможен только для свободных работников.

Покупка работников:

- за деньги;
- сначала можно покупать только работников 1 уровня;
- доступный уровень покупки зависит от максимального достигнутого уровня работника;
- если максимальный работник 1–3 уровня, можно покупать только 1 уровень;
- если достигнут 4 уровень, можно покупать 2 уровень;
- если достигнут 5 уровень, можно покупать 3 уровень;
- и так далее.

Правило:

```ts
maxPurchasableWorkerLevel = max(1, highestUnlockedWorkerLevel - 2);
```

## 9. Разработка клеток работниками

Игрок выбирает соседнюю закрытую клетку для разработки. Работник не стоит на закрытой клетке. Он занимает одну из соседних открытых достижимых клеток.

Если к целевой клетке есть несколько свободных соседних клеток, её могут одновременно разрабатывать несколько работников — максимум 4, по одному с каждой стороны.

Одна открытая клетка может быть занята только одним работником.

Скорость разработки клетки равна сумме скоростей всех работников, назначенных на неё.

Индивидуальный вклад работников в добычу не отслеживается. Важен только общий результат.

Перемещение работников мгновенное, если есть путь от входа уровня до позиции работника.

Правило переназначения работников во время активной смены пока не принято. Engine должен поддерживать это как конфиг:

```ts
allowWorkerReassignmentDuringShift: boolean;
```

## 10. Склад

Для каждого ресурса есть отдельные склады.

- Один склад хранит один тип ресурса.
- Для одного ресурса может быть несколько складов.
- Вместимость складов одного ресурса суммируется.
- Склады можно покупать.
- Склады можно улучшать.
- Если для ресурса нет свободной вместимости, этот ресурс не добывается.

Пример:

```ts
Storage {
  id: StorageId
  resourceId: ResourceId
  level: number
  capacity: number
  storedAmount: number
}
```

## 11. Заказы

Продажа ресурсов идёт через систему заказов.

Заказ может требовать один или несколько ресурсов.

```ts
Order {
  id: OrderId
  requirements: OrderRequirement[]
  rewardMoney: number
  state: OrderState
  expiresAtTick: number
  priority: number
}
```

```ts
OrderState =
  | "available"
  | "accepted"
  | "completed"
  | "expired"
  | "declined"
```

Игрок может отклонять заказы, которые не собирается выполнять.

Если игрок начал выполнять заказ, он обязан его завершить. Такой заказ нельзя отменить.

Если ресурсов хватает полностью, заказ выполняется сразу.

Если ресурсов не хватает, заказ можно принять в работу. В конце каждой смены ресурсы распределяются между активными заказами.

Если несколько заказов претендуют на один ресурс, нужно поддержать один из режимов:

```ts
OrderAllocationMode =
  | "priority_based"
  | "proportional"
```

Для MVP предпочтителен `priority_based`.

## 12. Фаза планирования

После завершения смены игра переходит в фазу планирования.

В фазе планирования доступны:

- покупка работников;
- merge работников;
- покупка складов;
- улучшение складов;
- принятие заказов;
- отклонение заказов;
- выставление приоритетов заказов;
- распределение работников;
- ручное сохранение;
- старт следующей смены.

Во время активной смены доступны:

- наблюдение;
- `read`;
- обычный `tick`;
- fast-forward до конца смены.

Покупки, merge, склады и работа с заказами во время активной смены запрещены.

## 13. Сохранение

Для MVP используется SQLite.

Сохранение не должно быть одним JSON blob. Данные должны храниться структурно.

Минимальные таблицы:

```sql
saves
levels
chunks
cells
cell_components
workers
storages
orders
order_requirements
```

Сохранение должно включать:

- seed-фразу;
- версию генератора;
- игровое время;
- все открытые уровни;
- все сгенерированные чанки;
- все клетки;
- компоненты клеток;
- видимость;
- достижимость;
- работников;
- склады;
- заказы;
- деньги;
- открытые ресурсы.

Persistence должен быть отдельным адаптером.

Engine не должен напрямую зависеть от SQLite.

Адаптер должен уметь:

- читать SQLite-сохранение;
- восстановить `EngineState`;
- создать engine из состояния;
- сохранить актуальный state engine обратно в SQLite.

## 14. Автосейвы

Сохранение выполняется:

- вручную по требованию игрока;
- автоматически после каждой смены.

Engine не пишет в SQLite сам. Он возвращает событие:

```ts
{ type: "autosave_requested", reason: "shift_completed" }
```

Application layer обрабатывает событие и вызывает persistence adapter.

## 15. Требования для TypeScript + Three.js варианта

### 15.1. Цель варианта

TypeScript + Three.js версия используется как web/desktop-friendly реализация с быстрой итерацией UI и визуализации.

### 15.2. Рекомендуемая структура проекта

```txt
/apps/web-three
  src/
    main.ts
    render/
    ui/
    input/
    application/

/packages/engine
  src/
    GameEngine.ts
    commands/
    queries/
    state/
    simulation/
    generation/
    economy/
    workers/
    orders/
    storage/
    time/

/packages/persistence-sqlite
  src/
    SqliteSaveAdapter.ts
    migrations/
    repositories/

/packages/shared
  src/
    ids.ts
    types.ts
    errors.ts
```

### 15.3. Engine

TypeScript engine должен быть независим от Three.js, DOM, canvas, React/Vue/Svelte и SQLite.

Engine package не должен импортировать:

- `three`;
- DOM API;
- UI-компоненты;
- SQLite-драйвер;
- browser storage API.

Engine предоставляет:

```ts
interface GameEngine {
  apply(command: EngineCommand): ApplyResult;
  read(query: EngineQuery): ReadResult;
  exportState(): EngineState;
}

class GameEngineFactory {
  static createNew(config: NewGameConfig): GameEngine;
  static createFromState(state: EngineState): GameEngine;
}
```

### 15.4. Render слой

Three.js слой:

- получает данные только через `engine.read`;
- отправляет действия только через `engine.apply`;
- не хранит авторитетную копию игрового состояния;
- может хранить только view-cache, meshes, camera state, selected cell, hover state.

Three.js не должен знать внутренние правила добычи, заказов, склада и генерации.

### 15.5. Persistence

SQLite adapter должен быть отдельным пакетом.

Для web-версии допустимые варианты:

- desktop shell через Tauri/Electron + нативный SQLite;
- browser SQLite через WASM/OPFS;
- временный dev-adapter, если он реализует тот же интерфейс.

Но доменная логика engine не должна зависеть от выбранного persistence runtime.

### 15.6. Тесты

Для TypeScript варианта обязательны:

- unit-тесты engine;
- тесты генератора;
- тесты расчёта tick;
- тесты складов;
- тесты заказов;
- тесты merge работников;
- тесты pathfinding/reachability;
- тесты видимости на радиус 5;
- тесты SQLite adapter на тестовой базе.

Рекомендуемые команды:

```bash
npm run lint
npm run format
npm run typecheck
npm run test
npm run build
```

или эквивалент через `pnpm`/`yarn`.

## 17. Требования для кодовых агентов

В репозитории должны быть файлы:

```txt
AGENTS.md
docs/architecture.md
docs/game-design.md
docs/engine-api.md
docs/persistence.md
```

`AGENTS.md` должен жёстко фиксировать:

1. Нельзя менять состояние игры вне engine.
2. UI/render работает только через `apply` и `read`.
3. Engine не зависит от render/UI.
4. Engine не зависит напрямую от SQLite.
5. SQLite доступен только через persistence adapter.
6. Все изменения должны сопровождаться тестами, если затронута логика.
7. Перед завершением задачи агент обязан запустить:

   - lint;
   - format;
   - typecheck/build;
   - tests.

8. Если все проверки прошли, агент должен сделать commit в текущую ветку.
9. Если проверки не прошли, commit запрещён.
10. Агент обязан явно указать, какие проверки запускались.

Для TypeScript:

```bash
npm run lint
npm run format
npm run typecheck
npm run test
npm run build
```

## 18. Открытые решения

Пока не зафиксированы:

1. Можно ли переназначать работников во время активной смены.
2. Должна ли добыча во время смены быть полностью locked-in или допускаются аварийные ручные вмешательства.
3. Точные численные коэффициенты:

   - скорость работников;
   - объём компонентов клеток;
   - цены заказов;
   - стоимость работников;
   - стоимость складов;
   - вместимость складов;
   - частота появления заказов;
   - лимиты пустых областей.

4. Конкретный UI-фреймворк для TypeScript-варианта.
5. Конкретный SQLite runtime для web/desktop TypeScript-варианта.
